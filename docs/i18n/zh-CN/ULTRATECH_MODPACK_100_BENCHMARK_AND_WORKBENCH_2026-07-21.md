## 基线结果转化为超技术规则

| 信号 | 超技术规则 |
| --- | --- |
| 当一个设施去除一个可以识别的旧瓶颈时, 长时间的阶段会感觉得到。 | 每个三阶段核心设施都获得生成的下个目标卡:老瓶颈,先起输入,持续受限,输出,恢复出站. |
| 食谱深度在形成分批,物流,回收,或平行决定时是有价值的. | 拒绝只重复手绘的食谱图层; 保留绑有图层 `IndustrialRoute` 和自动化阈值。 |
| 当第一个缺失的要求明显可见时,一个困难包仍然可以读取。 | 研究终端,扫描仪,和计算广播机必须报告相同的当前目标,缓存,提供FloPS,并屏蔽原因. |
| 资源压力必须有一个可再生或回收的答案。 | 扩展路由审计和机器状态,同时提供缺少的投入、能量、产出、断流和不稳定恢复指导。 |
| 当成熟的基础设施仍然相关时,后期游戏系统就会发挥作用。 | 空间和终端系统保留电力、冷却、后勤、研究、维修和回归之路的义务。 |

## 原思想积压

### P0 — 玩家方向

1. 从现有生成核心设施下一个目标卡
   `IndustrialRoute`, (中文(简体) ). `RecipeArchetype`和研究数据。
2. 为研究终端添加一个共享的玩家状态研究模型,
   扫描终端和计算广播机。
3. 提出每个机器块状态的直接回收行动:没有输入,
   没有能量,输出完全,断开的流量, 和不稳定的过程。

### P1——食谱和工厂可读性

1. 要求每一个新的专家食谱家族 指定其批量边界,
   第一个自动化点,主要消费者, 和回收出口。
2. 当食谱家族没有可见时添加路由- 审计警告
   消费者或未显示回收端点。
3. 使更换设施明确说明先前的容量、压力、
   冷却、纯度或物流限制,可以消除。

### P2 — 结束游戏连续性

1. 增加空间基础设施发射回程卡.
2. 对照成熟的动力、冷却、计算和
   所需回收费用,而不仅仅是最后设计费用。
3. 添加一个受限资源循环和一个受限资源的里程碑接受世界
   恢复-启动重启方案。

没有 P0/P1/P2 项自动更改游戏播放. 它成为执行
研究门、源、消费者、能源/氟化烃
边界、恢复退出、移徙影响和人工接受情况
定义。

## 工作台常规

1. 运行 `scripts/check-forge-env.ps1` 从安装的超技术技能。
2. 运行 `.\scripts\verify-dev-workstation.ps1`; 检查 Git 状态和稀有
   在判断缺失资源之前退出 。
3. 对于研究工作,验证目录,只实现外部
   学习登记。 对于游戏游戏工作,检查相邻的代码和已有
   DataGen在编辑前.
4. 运行 `runData` 食谱/资源/数据源更改;运行 `build` 用于源代码
   更改; 运行 `runClient` 用于GUI,机器,研究,或客户端行为.
5. 记录已更改的文件、命令退出代码、运行时间证据和剩余
   任务结果中的风险。

本地代理配置、 IDE 元数据以及 `runclient-*.log` 为:
只有工作站的文物。 他们被排斥在外 `.git/info/exclude`, (中文(简体) ).
没有承诺给GitHub或Gitee。

## 取样后的实施计划

### P0 — 在添加更多内容前恢复可辨别性

1. ** 客户端字体加载:** 审计现有的客户端资源路径和字体
   注册,只修补缺失的装入/装入链接,然后在
   `runClient`。 。 。 。
2. ** 一个研究状态: 制作研究台,空白知识书,
   学习知识书、扫描仪终端和研究终端读取
   写一个服务器授权研究记录。 书必须保留
   特定的研究密钥, 右键学习必须是一能的, 扫描仪
   提取必须添加可见的部分进展,而不是绕过一个门。
3. ** 反馈:** 披露相同的阻塞原因和下一步行动。
   研究台、扫描仪终端、研究终端和工厂
   (原始内容存档于2013-10-12). Recovery UI. 这是第一个得到双方支持的执行候选人
   研究基线和档案结构的
   配置/自动化支持。

接受:玩家可以在桌上研究,写出一本空白的书来学习.
一次通过右键点击,扫描到终端进度,然后从
缺少书本, 缺少电源/ FloPS, 完全输出, 或已学习状态
数据损失。

### P1——工厂进度合同

1. 为每个核心工业路线加上一个生成的下一个目标卡
   旧的瓶颈,第一次自动化输入,持续能量/FloPS限制,
   第一个消费者,恢复出口。
2. 延长食谱/程序审计,使新的专家家庭申报批次
   边界、自动化过渡、消费者和副产品/回收出口。
3. 添加暂停和恢复研究/工业工作的进度状态测试:
   缺少输入, 能量缺失, FloPS 缺失, 物流中断,
   完全输出。

接受:审计可以拒绝没有消费者或回收渠道的路线,
每个暂停状态都有玩家可见的理由加上无损的恢复
路径。

### P2——里程碑和结局连续性

1. 增加限制资源限制的里程碑接受设想,
   批量/平行扩展,并重新启动恢复启动。
2. 保持空间/末端游戏门与成熟的动力、冷却、物流和
   研究计算,而不是只计算出最后的工艺。
3. 在每次核定P1/P2变更后重新进行30个阶段/700小时的审计;不
   改变基线时间预算仅仅是因为样本库较大。

接受:所有六个里程碑都保留明确的恢复出口和
现有700小时/30相基准保持不变,除非另行核准
平衡变化提供证据和移徙说明。
