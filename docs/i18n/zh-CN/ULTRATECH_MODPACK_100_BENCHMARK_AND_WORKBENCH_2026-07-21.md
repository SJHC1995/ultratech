# UltraTech 100 包基准与可复现工作台

*2026-07-21 · 原创设计研究与开发基线*

## 不可妥协的基线

- Minecraft 1.20.1、Forge 47.4.4、位于 `C:\Java17` 的 Java 17，以及
  Parchment 2023.06.26-1.20.1。
- 保持 700 小时、30 阶段、六里程碑的进度预算。
- 外部整合包仅用于研究原创设计模式；不得复制配方、任务文本、资产、
  脚本、本地化、ID 或数值。

100 样本目录经过刻意分层：35 个 GregTech/长流程包、20 个工厂包、
15 个专家向跨模组包、10 个硬核生存包、10 个受限资源包和 10 个太空/
终局包。其机器可读来源是 `scripts/modpack-study-catalog.ps1`；外部
元数据和可选的稀疏源码 checkout 均留在本地外部研究目录。

初始执行保存了 13 个固定的公开来源版本和 87 份发行元数据档案。后者
有意保持初步性质：它们用于引导采样覆盖，但不被视为配方级证据。

专家向跨模组层包含用户提供的 MCMod 参考
`StellarLightTech`（`mcmod.cn/modpack/1347.html`）。该页面只用于高层
进度研究；其链接的网盘存档超出获准的 MCMod/GitHub/Modrinth 来源边界，
因此不会下载。

## 转化为 UltraTech 规则的基线结论

| 信号 | UltraTech 规则 |
| --- | --- |
| 当一座设施消除一个可识别的旧瓶颈时，长阶段才会显得值得。 | 每个跨三阶段的核心设施都生成“下一目标”卡：旧瓶颈、首次启动输入、持续约束、输出和回收出口。 |
| 当配方深度带来批处理、物流、回收或并行化决策时，它才有价值。 | 拒绝只会重复手搓的配方层；保留与 `IndustrialRoute` 和自动化阈值绑定的层。 |
| 当第一个缺失条件清晰可见时，高难整合包仍然可读。 | 研究终端、扫描终端和算力广播终端必须报告同一个当前目标、缓存、已供给 FloPS 与阻塞原因。 |
| 资源压力必须有可再生或回收的应对方案。 | 在路线审计和机器状态中扩展缺少输入、缺少能量、输出已满、流路中断与失稳恢复指引。 |
| 当成熟基础设施仍然相关时，后期系统才能成立。 | 太空和终端系统仍承担供电、冷却、物流、研究、维护和返程路径义务。 |

## 按优先级排列的原创创意待办

### P0 — 玩家方向

1. 由既有 `IndustrialRoute`、`RecipeArchetype` 和研究数据生成核心设施
   的“下一目标”卡。
2. 为研究终端、扫描终端与算力广播终端增加一套共用、面向玩家的研究状态
   模型。
3. 为每种机器方块状态提供直接恢复动作：无输入、无能量、输出已满、流路
   断开与过程失稳。

### P1 — 配方与工厂可读性

1. 要求每个新的专用配方族说明其批处理边界、首次自动化节点、主要消费者
   与回收出口。
2. 当配方族没有可见的首个消费者或没有展示回收终点时，增加路线审计警告。
3. 要求替代设施明确说明它消除了原先哪一项产能、压力、冷却、纯度或物流
   约束。

### P2 — 终局连续性

1. 为太空基础设施增加从发射到返程的路线卡。
2. 用成熟的供电、冷却、算力和回收要求审计终端建造，而不只看最终合成成本。
3. 为一个受限资源循环和一个回收循环重启场景增加里程碑验收世界。

任何 P0/P1/P2 项都不会自动改变游戏玩法。只有在定义其阶段、研究门槛、
来源、消费者、能量/FloPS 边界、回收出口、迁移影响和手动验收场景后，
它才会成为实施工作。

## 工作台例程

1. 从已安装的 UltraTech skill 运行 `scripts/check-forge-env.ps1`。
2. 运行 `.\scripts\verify-dev-workstation.ps1`；在判断资源缺失前检查
   Git 状态和稀疏 checkout。
3. 研究工作先验证目录，并只实体化外部研究注册表；玩法工作则应在编辑前
   检查相邻代码和既有 DataGen。
4. 配方/资源/DataGen 改动运行 `runData`；源码改动运行 `build`；GUI、
   机器、研究或客户端行为改动运行 `runClient`。
5. 在任务结果中记录改动文件、命令退出码、运行时证据和残余风险。

本地 agent 配置、IDE 元数据和 `runclient-*.log` 都是仅限工作站的产物。
它们通过 `.git/info/exclude` 排除，不会提交到 GitHub 或 Gitee。

## 采样后的实施计划

### P0 — 在添加更多内容前恢复可读性

1. **客户端字体加载：** 审计既有客户端资源路径和字体注册；只修复缺失的
   加载/绑定链接，再在 `runClient` 中验证。
2. **统一研究状态：** 让研究桌、空白知识书、已学习知识书、扫描终端和
   研究终端读写同一份服务器权威的研究记录。书必须保留具体研究键；
   右键学习必须幂等；扫描提取必须加入可见的部分进度，而不能绕过门槛。
3. **恢复反馈：** 在研究桌、扫描终端、研究终端和工厂恢复 UI 中公开同一
   阻塞原因和下一步动作。这是同时受到研究基线与存档结构中配置/自动化
   支持普遍性支撑的首个实施候选项。

验收：玩家能在研究桌研究、写出一本空白书、通过右键仅学习一次、将其
扫描为终端进度，并能在缺书、缺电/FloPS、输出已满或已学习状态下无数据
损失地恢复。

### P1 — 工厂进度契约

1. 为每条核心工业路线添加生成的“下一目标”卡，包含旧瓶颈、首次自动化
   输入、持续能量/FloPS 约束、首个消费者和回收出口。
2. 扩展配方/路线审计，使新的专用配方族声明批处理边界、自动化转换、
   消费者以及副产物/回收出口。
3. 为暂停和恢复的研究/工业工作增加进度状态测试：缺输入、缺能量、
   缺 FloPS、物流断开和输出已满。

验收：审计能够拒绝没有消费者或回收出口的路线；每个暂停状态都要有
玩家可见的原因和不破坏数据的恢复路径。

### P2 — 里程碑与终局连续性

1. 为受限资源启动、批处理/并行扩张和回收循环重启增加里程碑验收场景。
2. 将太空/终局门槛持续绑定到成熟的供电、冷却、物流和研究算力，而不只
   绑定一次最终合成。
3. 每次获准的 P1/P2 变更后，都重新执行 30 阶段/700 小时审计；不能仅因
   样本池扩大而修改基线时间预算。

验收：六个里程碑都保留明确的回收出口；既有 700 小时/30 阶段基线保持
不变，除非另有经批准的平衡改动同时提供证据和迁移说明。
