# UltraTech 算力请求身份化 - 实施与验收规划

## 目标

让每一项算力工作都可追溯、可拒绝、可重试，并让广播终端永远只拉起未被拉黑的请求 UUID。该规则适用于研究、蓝图、航天、无线能量、机器人、巨构和后续接入的所有算力机器。

## 已完成基础

1. `PlayerResearchSavedData` 数据版本升级到 `6`。
2. 研究请求和机器请求均使用八位 UUID。
3. 放置机器使用“机器类型 + 维度 + 坐标”作为稳定来源键。
4. 广播端按来源排序逐个做 120% 冗余额度检查。
5. 超额时仅拉黑当前请求 UUID；其余可承载请求继续广播。
6. 空请求立即清零面板需求与广播数值。
7. 旧匿名聚合请求在迁移时清除，避免存档遗留。

## 后续接入准则

任何新增的算力消费者必须满足以下接口契约：

1. 使用 `requestNetworkCompute(sourceId, amount)` 提交持续需求。
2. 使用稳定、可区分实例的 `sourceId`；方块实体必须包含维度和坐标。
3. 任务完成、取消、输入失效或不再缺算力时调用 `clearNetworkComputeRequest(sourceId)`。
4. 玩家显式点击“重试”“启动”“发射”或“重新解码”时调用 `restartNetworkComputeRequest(sourceId, amount)`。
5. 不允许恢复旧 UUID 黑名单，也不允许以每 tick 新 UUID 绕过限额。
6. 所有状态文本继续用 `FloPS`，不显示内部冗余计算单位。

## 验收场景

| 场景 | 预期 |
|---|---|
| 无处理器 | 请求等待，不产生永久黑名单；面板不显示历史需求 |
| 低阶处理器无法承载单请求 | 仅该 UUID 进入黑名单，广播重算其余请求 |
| 研究终端点击重试 | 新 UUID 生成，旧 UUID 仍保留在历史黑名单 |
| 两台同类机器不同坐标 | 各自有独立请求和独立黑名单 |
| 任务完成/机器停机 | 来源请求被清除，需求与广播归零 |
| 版本 5 存档加载 | 研究 UUID 保留，匿名旧队列清除 |
| 重启世界 | 版本 6 机器请求与 UUID 黑名单保持一致 |

## 自动校验

- 静态搜索不得存在无来源参数的 `requestNetworkCompute(...)` 调用。
- 编译通过 `compileJava` 与 `build`。
- 运行时检查推理终端面板：请求为 `0 FloPS` 时，需求也必须为 `0 FloPS`。
- 回归检查研究终端的拒绝文案、重试按钮与进度状态。

## 残余风险

机器请求由玩家网络缓存统一结算，因此同一玩家的大量自动化设备会在同一广播周期竞争处理器额度。当前以稳定来源排序保证结果可复现；后续如加入优先级 UI，应在请求记录中增加优先级字段，而不是重新引入匿名总量。
